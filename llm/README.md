# 一些基础概念
>来自AI

## 向量(vector)

文本在数学空间中的"坐标"，语义相近的文本，坐标距离就近；语义无关的文本，坐标距离就远。向量数据库就是专门高效查找"最近坐标"的数据库。

**数学定义：**
```
向量 = [v₁, v₂, v₃, ..., vₙ]

其中：
- n = 维度（dimension），常见 384、768、1024、1536、3072
- 每个 vᵢ 是一个浮点数，通常范围 [-1, 1] 或 [-∞, +∞]

示例：
"苹果" 的向量表示 ≈ [0.23, -0.87, 0.15]
"香蕉" 的向量表示 ≈ [0.25, -0.82, 0.18]
"汽车" 的向量表示 ≈ [-0.91, 0.34, -0.67]
```
**为什么文本要变成向量**
核心原因：计算机只能处理数字，不能直接理解文字。向量把语义信息编码成数字空间中的位置，使得：

| 特性              | 说明                       |
| --------------- | ------------------------ |
| **语义相似 = 距离相近** | "苹果"和"香蕉"的向量距离很近（都是水果）   |
| **语义无关 = 距离很远** | "苹果"和"汽车"的向量距离很远         |
| **可计算**         | 可以用数学公式（余弦相似度、欧氏距离）衡量相似性 |

**在RAG中的完整流程：**
```
1. 文档切分
   "这是一篇关于数据库配置的文档。首先你需要安装驱动..."
       ↓
   Chunk 1: "这是一篇关于数据库配置的文档"
   Chunk 2: "首先你需要安装驱动程序"
   Chunk 3: "然后配置连接字符串"

2. 向量化（每个 chunk 变成一个向量）
   Chunk 1 → [0.12, -0.34, 0.56, ...]  (768维)
   Chunk 2 → [0.15, -0.29, 0.61, ...]
   Chunk 3 → [0.18, -0.31, 0.58, ...]

3. 存入向量数据库
   ID | 向量 | 原始文本 | 元数据
   ---|------|---------|-------
   1  | [0.12, ...] | "这是一篇..." | {source: "db-guide.md"}
   2  | [0.15, ...] | "首先你需要..." | {source: "db-guide.md"}
   3  | [0.18, ...] | "然后配置..." | {source: "db-guide.md"}

4. 用户查询
   "怎么连数据库？"
       ↓
   嵌入模型编码 → [0.14, -0.32, 0.57, ...]
       ↓
   向量数据库做 ANN 搜索（找最近邻）
       ↓
   返回 Top 3 最相似的 chunks
       ↓
   拼接给 LLM 生成回答
```

**举个例子：**

| 层级             | 含义                            | 换算关系                  |
| :-------------- | :----------------------------- | :--------------------- |
| **原始文档**       | 你上传的完整文件（PDF、Word、Markdown 等） | 1 个文件                 |
| **文档块（Chunk）** | 把文档切分成的小片段（通常 200-1000 字）     | 1 个文件 ≈ 5-50 个 chunks |
| **向量（Vector）** | 每个 chunk 经嵌入模型转换后的高维数组        | 1 chunk = 1 个向量       |

**换算示例**
```
1 本 300 页手册
    ↓ 按 500 字/块切分
≈ 300 个 chunks（假设每页约 500 字）
    ↓ 经嵌入模型编码
= 300 个向量（768 维或 1536 维）
```
那么百万级向量约等于3333-33333份文档

**关键特性：**

| 特性       | 说明                          |
| :-------- | :--------------------------- |
| **高维**   | 通常 384-3072 维，人类无法直观理解      |
| **稠密**   | 大部分维度都有非零值（vs 稀疏向量如 TF-IDF） |
| **语义编码** | 相似语义的文本，向量在空间中的位置相近         |
| **固定长度** | 同一模型输出的所有向量维度相同             |
| **可比较**  | 只能用同一模型生成的向量互相比较            |

## 检索增强生成(Retrieval-Augmented Generation,AKA: RAG)

**它要解决问什么问题？**

| 问题           | 表现                |
| :------------ | :----------------- |
| **幻觉**       | 编造不存在的事实、假数据、假引用  |
| **知识截止**     | 模型训练后发生的事件完全不知道   |
| **无法访问私有数据** | 公司内部文档、个人笔记、私有数据库 |
| **无法溯源**     | 不知道回答来自哪里，无法验证    |

RAG怎么做:
```
传统 LLM：
用户提问 → LLM 凭"记忆"生成回答 → 可能胡说

RAG：
用户提问 → 先检索相关资料 → 把资料+问题一起给 LLM → 基于资料生成回答
              ↑___________________________↓
                        可溯源、可验证
```

完整流程：
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   文档输入    │────►│   预处理     │────►│   向量化    │
│ (PDF/Word/   │     │ (切分/清洗/   │     │ (嵌入模型    │
│  Markdown/   │     │  去重/元数据) │     │  编码为向量) │
│  网页/数据库) │     │             │     │             │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
                                        ┌─────────────┐
                                        │  向量数据库   │
                                        │ (存储+索引   │
                                        │  所有向量)    │
                                        └──────┬──────┘
                                               │
┌─────────────┐     ┌─────────────┐     ┌────┴────────┐
│   用户提问    │────►│   查询向量化  │────►│  相似度检索   │
│ "如何配置     │     │ (同一嵌入模型  │     │ (ANN 近似    │
│  数据库？"    │     │  编码问题)    │     │  最近邻搜索)  │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
                                        ┌─────────────┐
                                        │  返回 Top-K  │
                                        │  相关文档片段 │
                                        │ (chunks)     │
                                        └──────┬──────┘
                                               │
                                               ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   LLM 生成   │◄────│   构建提示词   │◄────│  拼接上下文   │
│  最终回答    │     │ (问题+检索到的  │     │ (文档片段+    │
│             │     │  文档片段)     │     │  元数据)     │
└─────────────┘     └─────────────┘     └─────────────┘
```
**RAG vs 微调（Fine-tuning）**

| 对比       | RAG         | 微调        |
| :-------- | :----------- | :--------- |
| **原理**   | 检索外部资料 + 生成 | 修改模型参数    |
| **成本**   | 低（查数据库）     | 高（训练 GPU） |
| **更新速度** | 实时（改文档即可）   | 慢（需重新训练）  |
| **适合数据** | 频繁变动、大量文档   | 固定知识、特定风格 |
| **可溯源**  | ✅ 能指出来源     | ❌ 无法溯源    |
| **隐私**   | ✅ 数据不外泄     | ❌ 需上传数据训练 |
| **幻觉控制** | ✅ 基于资料回答    | ⚠️ 仍可能幻觉  |
> 其实可以先RAG,再微调

变种们：

| 类型               | 特点            | 适用场景   |
| :---------------- | :------------- | :------ |
| **Naive RAG**    | 基础版：一次检索 + 生成 | 简单问答   |
| **Advanced RAG** | 查询重写、重排序、多路召回 | 复杂查询   |
| **Modular RAG**  | 路由、查询分解、多跳推理  | 多步骤任务  |
| **Agentic RAG**  | LLM 自主决定检索策略  | 深度研究   |
| **Graph RAG**    | 结合知识图谱，理解实体关系 | 复杂关联查询 |
